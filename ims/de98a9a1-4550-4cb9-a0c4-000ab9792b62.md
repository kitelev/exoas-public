# RFC-020: Alpha Tester Onboarding

**Status:** DRAFT v3
**Author:** ExoAssistant
**Date:** 2026-04-10 (v3: post multi-agent review)

**Repo:** kitelev/exocortex

## 1. Мотивация

Exocortex завершил инициативу "Динамическое конфигурирование" (RFC-009 → RFC-019). Плагин стал vault-driven rendering engine — все команды, кнопки, формы определяются markdown-файлами.

**Сценарий:** Пользователь получает ссылку на `github.com/kitelev/exocortex`, устанавливает и настраивает EMS (Areas, Projects, Tasks, управление через кнопки и Command Palette).

**Проблема:** Этот сценарий невозможен в текущем состоянии из-за критических блокеров.

**Target persona:** Технически грамотный пользователь Obsidian, знающий Git/npm, desktop-only. Не beginner, не mobile user. Расширение persona — в будущих версиях.

## 2. Архитектурный контекст: почему проблема существует

### Инверсия Obsidian-паттерна

В экосистеме Obsidian **ни один community plugin** не требует копирования сотен файлов в vault. Стандарт:

| Plugin | Файлов для старта | Delivery |
|--------|-------------------|----------|
| Dataview | 0 | Query language, works instantly |
| Templater | 0 | Settings → folder picker |
| Tasks | 0 | Inline syntax, zero setup |
| Kanban | 0 | Creates board from new note |
| **Exocortex** | **~418** | Требует внешней установки |

Exocortex инвертирует convention: 188 `exocmd/` файлов — это **plugin configuration layer** (определения кнопок, preconditions, groundings), но хранится в vault как **user data**. Это корневая причина всех onboarding-проблем.

### Текущее состояние

| Слой | Файлов | Назначение | Нужен для EMS? |
|------|--------|------------|----------------|
| `exo/` | 85 | Core ontology (classes, properties) | Частично (TBD Phase 0) |
| `ems/` | 134 | EMS classes, statuses, properties | Частично (TBD Phase 0) |
| `exocmd/` | 188 | Button definitions, groundings | Все (CommandResolver requires) |
| `pn/` | 3 | Daily Notes class | Да (Daily Planning) |
| Templates | 8 | Templater templates | Опционально |
| **Итого** | **~418** | | **TBD** |

**Как плагин обнаруживает файлы:** `VaultRDFIndexer` → `NoteToRDFConverter.convertVault()` → `app.vault.getMarkdownFiles()` — сканирует **все** .md файлы в vault. Путь произвольный, `03 Knowledge/` — конвенция, не хардкод.

## 4. Prerequisite: Fix Bug #2688

> **Без этого alpha onboarding невозможен вне зависимости от delivery mechanism.**

**Проблема:** E2E тест `dynamic-command-buttons-render.spec.ts` помечен `test.fixme`:

```
// FIXME(#2688): CommandResolver finds 3 commands (confirmed by diagnostics),
// but buttons don't appear in DOM after refreshLayout().
```

- CommandResolver **находит** команды в triple store
- Кнопки **не рендерятся** в DOM в Docker E2E environment
- Тест пропущен — CI не ловит регрессии в button pipeline

**Impact:** Если buttons не рендерятся в Docker E2E → возможна аналогичная проблема в production Obsidian. Alpha-тестер увидит пустой button bar даже с полной ontology.

**Action:** Исследовать и починить #2688. Снять `test.fixme`. Подтвердить что buttons рендерятся и кликабельны в E2E.

**Effort:** 2-4h (debugging `DynamicCommandButtonGroupBuilder.build()` — extractSubjectIRI/extractAssetClass в Docker metadata format)

## 5. Phase 0: Верификация (сфокусированная)

> **Code analysis уже отвечает на главные вопросы. Phase 0 верифицирует runtime-поведение.**

### Что известно из кода (не требует эксперимента)

- `CommandResolver.findBindings()` queries `rdf:type = exocmd:CommandBinding` → без exocmd/ = 0 buttons
- `valueToClassURI("ems__Task")` расширяется через `NAMESPACE_MAP` **без file lookup** → ems/ class definitions НЕ обязательны для targeting
- Все exocmd/ файлы **обязательны** (CommandBinding → Command → Grounding → Precondition chain)

### Что требует runtime-верификации (3 вопроса)

| # | Вопрос | Как проверить | Влияние |
|---|--------|---------------|---------|
| 1 | Оба формата frontmatter работают? (`ems__Area` vs `- "[[1b20a8f0-d745-4e93-91db-4531b3df120e]]"`) | Создать note обоими способами, проверить layout | Определяет формат в guide |
| 2 | VaultRDFIndexer подхватывает новые файлы через file watcher? | Скопировать ontology в vault без restart, проверить buttons | Если нет — это bug, не design |
| 3 | Нужны ли ems/ status files для SPARQL ASK preconditions? | Добавить только exocmd/ + pn/, кликнуть status button | Определяет MVO |

**Effort:** 15-20 минут в чистом vault.

**Важно (file watcher):** Если VaultRDFIndexer не реагирует на `vault.on('create')` — это **регрессия**, не ожидаемое поведение. Obsidian API предоставляет file watcher. Все major plugins (Dataview, Tasks) его используют. Fix = wiring event, не "документировать restart".

## 6. Блокеры (prioritized)

### P0 — Без этого onboarding невозможен

#### P0-prerequisite: Fix #2688 (buttons don't render in DOM)

См. секцию 4.

#### P0-1: Scaffold command — запись ontology из plugin bundle

**Проблема:** 418 vault files существуют только в dev vault.

**Решение: Option D — scaffold command** (primary, не future)

Обоснование:
- Единственный вариант, не требующий выхода из Obsidian
- 4 из 4 ревьюеров рекомендовали как primary
- Реализация проще чем кажется: build step (`glob + JSON.stringify`) + `vault.create()` on first run
- ZIP/degit/git clone — fallback для power users, не primary path

**Реализация:**

```
Build time:
  glob("ontology/**/*.md") → JSON snapshot → embedded in main.js

Runtime (first enable or Cmd+P → "Exocortex: Initialize Starter Kit"):
  1. Check if ontology files exist (any exocmd__CommandBinding in vault?)
  2. If not → write all files via app.vault.create()
  3. Trigger VaultRDFIndexer re-index
  4. Show Notice("Exocortex Starter Kit installed. Switch to Reading Mode.")
```

**Fallback для manual install (power users):**
- ZIP в GitHub Release (все ontology файлы)
- `npx degit kitelev/exocortex-ontology` → vault

**Effort:** 4-6h (build script + runtime scaffold + tests)

#### P0-2: Getting-Started.md — installation path

**Fix:** BRAT = единственный путь. Убрать Community Plugins, убрать git clone.

#### P0-3: Getting-Started.md — ontology install step

**Fix:** Добавить Step 2.5: "Initialize Starter Kit" (scaffold command) или manual install.

#### P0-4: First-run experience

**Проблема:** Plugin на first enable **молча загружается**. Нет notice, нет welcome.

**Fix (2h):**

```typescript
// ExocortexPlugin.onload()
if (!this.hasOntologyFiles()) {
  new Notice(
    "Exocortex: Run 'Initialize Starter Kit' from Command Palette " +
    "to set up your knowledge base.",
    15000
  );
}
```

Проверка: `hasOntologyFiles()` = есть ли хотя бы один `exocmd__CommandBinding` triple в store.

#### P0-5: `isDesktopOnly: true` для alpha

**Проблема:** `manifest.json` заявляет mobile support (`"isDesktopOnly": false`), но ни один delivery path (scaffold TBD, ZIP, degit, git clone) не работает на iOS/Android без terminal.

**Fix:** `"isDesktopOnly": true` до реализации mobile-ready scaffold command.

### P1 — Guide вводит в заблуждение

#### P1-1: Неправильные property names

| В guide | Каноническое | Тип |
|---------|-------------|-----|
| `ems__Effort_project` | `ems__Effort_parent` | Legacy alias, нет definition |
| `ems__Effort_scheduled_start_date` | `ems__Effort_plannedStartTimestamp` | Несуществующее |

#### P1-2: Frontmatter format — документировать canonical

Два стиля в разных docs. Верифицировать в Phase 0, документировать один canonical формат.

#### P1-3: Audit dead links в Getting-Started.md

"Next Steps" ссылается на: `workflows/Task-Workflow.md`, `workflows/Daily-Planning.md`, `workflows/Area-Organization.md`, `Plugin-Commands.md`, `Troubleshooting.md`, `Performance-Guide.md#mobile-optimization`. Проверить существование, удалить/пометить несуществующие.

#### P1-4: E2E test vault — включить exocmd/ файлы

**Факт (уточнение v2→v3):** Test vault содержит 12 exocmd файлов (3 команды × 4 файла) в `03 Knowledge/commands/`. НО DOM-render тест `test.fixme` (#2688). После fix #2688 — update test vault с MVO файлами.

#### P1-5: `resolveFilePath()` — O(n) scan на каждый button click

`ServiceRegistryPopulator.ts:284-296` — при каждом клике кнопки сканирует ALL markdown files для IRI → path resolution. В vault 5000 файлов = 200-500ms latency.

**Fix:** UID→path index в VaultRDFIndexer (already processes all files). Update в existing handlers `updateFile()/removeFile()/renameFile()`.

### P2 — Улучшает опыт

#### P2-1: README Quick Start → Getting Started link + Reading Mode note

#### P2-2: service_call health-check at startup

`GroundingExecutor` использует `targetProperty` как serviceId. 15 hardcoded IDs в `ServiceRegistryPopulator`. Drift = silent failure.

**Fix:** На startup валидировать все `service_call` groundings в vault против registered services. Warning notice при mismatch.

#### P2-3: Repo visual identity

Root-level: AGENTS.md, PATTERNS.md, TEMPLATES.md, WORKTREE_COORDINATION.md. Первое впечатление = "AI development lab". Для alpha: переместить dev-файлы в `docs/dev/` или `.github/`.

## 9. Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| #2688 deeper than expected (not just Docker) | Alpha buttons broken | Prioritize fix, test on real Obsidian desktop before alpha |
| Scaffold command bundles stale ontology | Buttons mismatch plugin version | `exocmd__minPluginVersion` property in ontology manifest; startup validation |
| Plugin bundle size increase (~418 files as JSON) | Larger main.js | Measure: if > 500KB overhead, consider lazy loading or separate asset file |
| User modifies scaffolded files → update conflict | Broken buttons after scaffold re-run | Scaffold skips existing files; "Reset to defaults" command for full override |
| `resolveFilePath()` O(n) degrades on large vaults | Slow button clicks | P1-5 UID→path index (not blocking alpha, but blocking production) |
| File watcher doesn't trigger re-index (Phase 0 Q2) | Requires restart after scaffold | Wire `vault.on('create')` in VaultRDFIndexer if missing |
| service_call serviceId drift between plugin/ontology versions | Silent button failure | P2-2 startup health-check |
